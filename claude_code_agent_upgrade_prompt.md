## 1. 배경 — 무엇을 왜 하는가

이 프로젝트는 사용자가 아이디어를 내고 코딩 에이전트(너 = Claude Code, 그리고 Antigravity)가
구현하는 방식으로 발전해 왔다. 누적된 문제는 **에이전트의 토큰 낭비**와 **"고치면 다른 게 깨짐"**이다.
사용자는 이를 막으려고 수동 체계(`_STATUS.md`·`DEBUG_JOURNAL.md`·`CODEMAP.md`·회귀게이트·Dev Mode)를
이미 구축했다. **이번 작업의 목표는 그 수동 체계를 "대체"하는 게 아니라, 약점 두 곳을 자동화로 보강**하는 것이다:

- (A) **검증이 에이전트의 성실성에 의존한다** → Hooks로 코드 레벨 강제.
- (B) **교훈 기록(RECORD)을 수동으로 한다** → 교정사항 자동 포착 **및 정본 문서 자동 반영**(단, 안전장치 3종으로 가드).

조사·타당성 평가 결론은 "기존 수동 체계가 이미 업계 베스트프랙티스의 수동판이므로, 통째 도입이 아니라
**기존 게이트/문서에 얇게 얹는 보강**만 한다"였다. 아래 등급은 그 결론을 반영한다.

## 2. 도입 등급 (이 순서대로, 하나씩 — 등급 간 사용자 승인 필수)

### ⚙️ 자동 문서수정 안전장치 3종 (자동 도구가 정본 문서를 수정하기 전 반드시 갖출 것)

사용자는 claude-reflect/자기개선 스킬이 `CLAUDE.md`·`_STATUS.md`·트랩 문서를 **직접 자동 수정**하도록 허용했다.
편의는 크지만 이 프로젝트 최대 리스크인 doc drift(저확신 교훈이 고밀도 정본을 오염)를 부를 수 있으므로,
**아래 3종을 자동수정 파이프라인에 반드시 내장**하라(계획서에 구체 설계 포함):

1. **백업/되돌리기:** 자동 도구가 정본 문서를 수정하기 **직전에** 원본을 타임스탬프 백업
   (`<file>_autoreflect_bak_YYYYMMDD_HHMM.md` 또는 `changes_history/_autoreflect/`). git이 없으므로 이 백업이 유일한 롤백 수단이다.
   사용자가 "방금 자동수정 되돌려"라고 하면 즉시 복원 가능해야 한다.
2. **정합성 게이트:** 자동수정 **직후** `uv run scripts/smoke_check.py`(+필요시 `health_check.py`)를 실행해
   문서-코드 정합·렌더 무결을 확인. **게이트 실패 시 자동수정을 자동 롤백**(1의 백업으로 복원)하고 사용자에게 경고.
3. **변경 출처 로그:** 모든 자동수정은 "어떤 도구가 / 어떤 문장을 / 왜(어느 세션의 어떤 교정에서) / 확신도 몇으로" 추가했는지를
   `changes_history/_autoreflect_log.md`에 append. 확신도 임계값(예: ≥0.85)을 **계획서에서 정하고**, 그 미만은 정본에 직접 쓰지 말고
   기존 `dev_notes/` 큐로 보내 사람이 검토하게 하라(고확신=자동반영, 저확신=큐, 의 2단 게이팅).

> 즉, "자동 직접수정"은 허용하되 **무방비가 아니라 백업·검증·로그·확신도 게이팅이 걸린 자동수정**이어야 한다.
> 이 3종이 없으면 자동수정 기능을 켜지 마라.



### 🟢 TIER 1 — 즉시 도입 권장 (효과 확실·위험 낮음)

#### 1-A. Claude Code Hooks → 회귀 게이트 자동 강제
- **목적:** 백엔드/템플릿 파일이 수정될 때마다 `smoke_check.py`를 **에이전트 의지와 무관하게** 자동 실행.
  지금은 "지침으로 부탁"이라 가끔 누락된다(과거 changes_73이 `py_compile`만 하고 verified로 기록해 앱을 죽인 전례).
- **구현 방향(계획서에 구체화할 것):**
  - `.claude/settings.json`을 **새로 생성**(기존 `.claude/launch.json`은 건드리지 말 것).
  - **`PostToolUse` 훅**: `Edit`/`Write` 매처에 한해, 변경 파일이 `scripts/*.py`(특히
    `market_dashboard3_realtime.py`·`ui_templates.py`·`pure_helpers.py`)이면 `uv run scripts/smoke_check.py`를
    실행하고, 비-0 종료 시 결과를 에이전트에 피드백.
  - **성능 가드(중요):** smoke_check가 무거우면 매 편집마다 도는 게 과하다. **debounce 또는 "세션 종료(Stop) 훅에서 1회"**
    중 어느 쪽이 토큰·시간 효율이 나은지 계획서에서 비교해 제안하라. 토큰 절감이 이 프로젝트의 목표이므로,
    훅이 오히려 비용을 키우지 않도록 설계하라.
  - **`UserPromptSubmit` 또는 `SessionStart` 훅**(선택): `_STATUS.md`의 Active Traps 한 줄 요약 또는
    `health_check.py` 결과를 세션 시작 시 주입 → 매번 큰 파일을 다시 읽는 토큰 낭비 감소.
- **검증:** 의도적으로 사소한 마크업을 바꿔 훅이 실제로 smoke_check를 돌리는지 관찰(로그/출력 인용).
  훅 도입 자체는 코드 로직 불변이지만, settings 변경이므로 앱 기동 1회로 부작용 없음 확인.
- **참고 레포(읽기용, 통째 복붙 금지):** Claude Code 공식 Hooks 문서.

#### 1-B. claude-reflect → 교정사항 자동 포착(반영은 사람 승인)
- **레포:** `https://github.com/BayramAnnakov/claude-reflect`
- **목적:** 세션 중 사용자가 "아니 X 말고 Y로"처럼 교정하면 자동 포착해 확신도 점수를 매기고,
  `CLAUDE.md`/`AGENTS.md`/`_STATUS.md`에 **직접 자동 반영**한다(사용자 결정). 사용자의 수동 RECORD를 자동화.
- **반영 정책 (사용자 승인됨):** 정본 문서 **자동 직접수정 허용**. 단 §2의 **자동 문서수정 안전장치 3종을 반드시 적용**한다:
  - **2단 확신도 게이팅:** 고확신(예: ≥0.85)만 정본에 직접 자동 기록, 그 미만은 기존 `dev_notes/` 큐로 보내 사람이 검토.
    (임계값은 계획서에서 확정. 이 프로젝트 정본은 고밀도라 임계값을 보수적으로 잡아라.)
  - **수정 전 백업 + 수정 후 `smoke_check` 정합성 게이트 + 실패 시 자동 롤백 + `_autoreflect_log.md` 출처 로그**(3종 전부).
  - 기존 `dev_notes/` 큐 워크플로(캡처→사용자확인→`done/` 이관)와 **개념을 통일**하라 — 저확신 큐는 dev_notes로 흡수하고,
    새 평행 체계를 난립시키지 말 것.
  - 정본을 망가뜨리지 않도록, 자동수정은 **기존 섹션 구조를 보존**(예: `_STATUS.md`는 Active Traps 섹션 끝에 append,
    표 형식 유지)하고 **문서 전체를 재작성하지 마라.**
- **구현 방향:** 레포의 메커니즘을 먼저 읽고, 이 프로젝트에 **얹을 수 있는 최소 형태**(가능하면 Stop 훅 +
  안전장치 래퍼)만 채택. 도구가 정본 직접수정 + 확신도 게이팅 + 백업을 깔끔히 지원하지 않으면,
  **동등 동작을 경량 자체 스크립트로 구현**(reflect는 포착만, 안전장치·반영은 자체 래퍼)하는 안을 계획서에 비교 제시하고 그쪽을 우선하라.
- **검증:** 모의 고확신 교정 → 정본에 안전하게 자동 반영되고 `_autoreflect_log.md`에 출처가 남는지 확인.
  모의 저확신 교정 → `dev_notes/` 큐로만 가는지 확인. 그리고 **일부러 정합성을 깨는 수정을 유도해 게이트가 자동 롤백하는지**(가장 중요) 확인.

2. **PLAN** — `application_build/changes_history/`에 다음 번호의 `changes_X_plan_agent_upgrade.md`를 작성:
   - 각 TIER 항목별: 도입 방식, 정확히 어떤 파일/설정을 만들/바꿀지, 검증 방법(어떤 게이트로 무엇을 관찰할지),
     롤백 방법, 트레이드오프.
   - **TIER 1만 "이번에 구현", TIER 2/3은 "제안·시범/보류"로 명확히 구분.**
   - 토큰 효율 관점의 설계 판단(특히 1-A 훅 빈도)을 반드시 포함.
3. **STOP & 승인 요청** — 계획서를 사용자에게 제시하고 **승인을 기다려라. 승인 전 코드/설정 변경 금지.**
4. **ACT (승인 후)** — TIER 1-A → 1-B 순서로 **하나씩** 구현. 각 항목 후 즉시 VERIFY.
5. **VERIFY** — 각 도입마다 관찰 기반 검증. 구조/설정 변경이면 `uv run scripts/smoke_check.py` → `SMOKE PASS ✓` 인용.
   훅이 의도대로 동작하는지 실제 트리거시켜 로그로 증명.
6. **RECORD** — `changes_X_*.md`(영어) 작성 + `_STATUS.md` 기능표/트랩 갱신. 비자명한 오류는 `docs/DEBUG_JOURNAL.md`에 append.
   구조가 바뀌었으면 `python3 scripts/gen_codemap.py` 재생성 + 루트 `CLAUDE.md`/`_STATUS.md` 동기화(§12).
   **새 트랩(예: "reflect 자동 문서수정은 고확신만 정본 직접반영·저확신은 dev_notes 큐, 수정 전 백업+후 smoke_check 게이트 필수")
   를 `_STATUS.md` Active Traps에 등재.**

## 4. 한 문장 요약 (이걸 잊지 마라)

> 기존 수동 체계를 갈아엎지 말고, **검증 강제(Hooks)** 와 **교정 자동 포착·정본 자동반영(reflect)** 두 곳만 얇게 보강하라.
> 정본 문서 자동수정은 허용하되 **반드시 안전장치 3종(백업·정합성 게이트·출처로그) + 확신도 2단 게이팅**을 걸어라 —
> 무방비 자동수정은 doc drift로 직결된다. 나머지는 시범/참고에 그치고,
> **계획 → 승인 → 하나씩 구현 → 게이트 검증 → 기록**의 순서를 어기지 마라. doc drift와 무단 대수술이 이 프로젝트 최대의 적이다.
