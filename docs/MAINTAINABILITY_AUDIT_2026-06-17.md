## 2. 파이썬 런타임 효율 — 대체로 양호 (과잉수정 불필요)

1. **정적 페이지는 모듈 로드 때 한 번만 조립**된다(`.replace` 가 module-level). 요청당 70KB 문자열을
   다시 깁지 않는다 → OK. 동적 리포트만 `_inject_*` 로 요청당 주입(불가피).
2. **캐시가 두텁다**: parquet 디스크 캐시 + TTL dict 캐시 155곳 + `_RESULT` RAM 캐시. 재조회는 거의 무료.
3. **SSE 를 위해 `threaded=True`** 가 켜져 있어 긴 LLM 스트리밍 중에도 `/__ping` 을 받는다(트랩#9).
4. 사소한 개선 여지(후순위, 단일 사용자라 영향 작음):
   - 동기 `httpx.get/post` 42곳 + `asyncio.run` 15곳이 혼재. 단일 사용자엔 무해하나, 한 요청에서 여러
     외부호출을 직렬로 하면 그 요청만 느려진다. 이미 `asyncio.gather` 를 쓰는 핫패스는 괜찮다.
   - `lru_cache` 0개 — 순수함수 캐싱은 직접 만든 TTL dict 로 대체(일관적이라 문제는 아님).

> 즉, "파이썬이 비효율적이라 느리다"는 아니다. 병목은 외부 API 지연이고 캐시로 잘 가려져 있다.

## 3. AI 에이전트 유지보수성 — 여기가 진짜 문제

### 🔴 3-1. 단일 13k줄·736KB 파일 = 컨텍스트 윈도 초과
- 파일 전체 ≈ **18만 토큰**(chars/4) → 대부분 모델의 한 컨텍스트에 **안 들어간다**. 에이전트는 어쩔 수
  없이 grep/부분 Read 로 *파일 일부만* 보고 편집한다 → 다른 구역과의 결합(예: `_inject_*` 주입 순서,
  `class="tab-btn"` 카운팅)을 못 보고 **조용한 회귀**를 낸다.
- **이번 세션의 실증**: changes_73 의 파일 이동이 라이브 빌더의 절대 import 를 깨 **앱이 아예 안 떴는데**도
  `py_compile`+스텁테스트만으로 "verified" 처리됐다(→ changes_74 로 수정). 큰 파일일수록 이런 누락이 잦다.

### 🔴 3-2. 파일의 절반이 마크업 문자열 → Python 을 찾는 토큰의 50% 낭비
- 에이전트가 파이썬 함수 하나를 grep 해도 341K자의 HTML/CSS/JS 를 헤집고 지나가야 한다.
- 템플릿을 `templates/*.html`(또는 `_templates.py`)로 빼면 Python 파일이 ~6–7k줄로 **반토막** 나고,
  마크업은 따로 하이라이트/린트 가능. 단, `.replace()` 앵커 136곳이 정확한 문자열 레이아웃에 의존하므로
  **Tier-L(계획 우선) 작업**이다 — 이번 세션에선 실행하지 않음(아래 권고 참조).

### 🟡 3-3. 내비게이션 스캐폴딩 부재
- 섹션 배너 주석 0개, 모듈 상단 목차 docstring 없음 → 에이전트가 "AI 엔드포인트 어디 / 실시간 어디"를
  빠르게 못 짚는다. **→ 이번 세션에 `docs/CODEMAP.md`(라인맵 인덱스) 신설로 보완**(아래 4).

### 🟡 3-4. 문자열 앵커 주입의 취약성
- `_inject_*` 8함수가 `</nav>`·`<footer`·`tab-btn` 카운팅으로 빌드된 HTML 을 사후 수술. 원본 빌더가
  앵커를 바꾸면 **무에러 누락**. changes_72 가 경고 로깅을 추가했지만 근본은 전용 주석 앵커(`<!--KMKT_M4_SLOT-->`).

### 🟡 3-5. 검증 신뢰도 (TECH_REVIEW #1·#2 재확인)
- 자동 테스트는 순수함수 26개뿐. 라우트/주입/스케일 변환은 사람이 눈으로 잡아야 한다.
- `py_compile = verified` 오기록이 반복(changes_2·13·28·38·73). **import 경로/라우트를 건드리면 반드시
  실제 import·앱 기동으로 검증**해야 한다(이번 changes_74 의 교훈, _STATUS.md 트랩#38).
