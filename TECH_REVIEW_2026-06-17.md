## 총평

1인 대학생 개발자가 AI 에이전트와 협업해 만든 결과물이라고 보기 어려울 만큼 **기능 폭과 데이터 통합 깊이가 넓다**. KIS·DART·KRX·네이버·FRED·Finnhub·FMP·Polygon·Gemini·로컬 LLM까지 10개 이상의 소스를 묶었고, 캐시·비동기 병렬·SSE 스트리밍·WebGL 3D 같은 고급 기법이 실제로 동작하는 형태로 들어가 있다. 동시에 이 프로젝트는 **전형적인 "AI 에이전트 단일 거대 파일" 안티패턴의 교과서**이기도 하다. 핵심 백엔드 1개 파일이 13,041줄·71개 라우트·259개 함수·142개 모듈 전역 상수를 단독으로 떠받치고, 화면은 32만 자에 달하는 HTML 문자열을 파이썬 안에 박아 넣은 뒤 `.replace()` 86회·정규식 46회로 꿰매는 구조다. 가장 큰 리스크는 코드 품질 자체보다 **검증 체계의 신뢰도**다. 자동화 테스트가 사실상 0이고, `_STATUS.md` 기준 25개 기능이 `❓ 미검증`으로 누적돼 있으며, 과거에 "고쳤다"고 기록했지만 실제로는 깨져 있던 사례(changes_2·6·13·28·38)가 반복적으로 발생해 로그 자체에 교정 기록(Correction log)이 따로 존재한다. 즉 **"선언된 상태"와 "실제 동작"의 간극이 이 프로젝트의 1순위 관리 대상**이다.

## 잘 된 점

**1. AI 협업 거버넌스 문서 체계가 이례적으로 성숙하다.**
`application_build/CLAUDE.md`의 `READ → DIAGNOSE → ACT → VERIFY → RECORD` 운영 루프, Task Tier(N/S/L) 분류, immutable changes 로그 + 살아있는 `_STATUS.md` 분리는 시니어 팀의 런북 수준이다. 특히 `_STATUS.md`의 **Correction log**(과거 잘못 기록된 항목을 명시적으로 추적)와 30여 개의 **Active Traps**(SSE는 generator 밖에서 `request` 읽기 등 hard-won 함정)는, 컨텍스트가 매번 초기화되는 AI 에이전트 환경에서 실수 재발을 막는 가장 현실적인 장치다. 이 문서 자산이 코드의 약점을 상당 부분 상쇄하고 있다.

**2. 함정을 알고 회피한 흔적이 코드에 실제로 남아 있다.**
trap #2(SSE)를 검증해 보면 `/api/llm_ask` 등에서 `request.args.get(...)`를 generator 함수 `_synth()` 정의 **이전**에 읽고 클로저로 넘긴다(4249~4256줄). bare `except:`가 0건이고 전부 `except Exception`(172건)으로 잡는 점, 모듈 전역 `global` 키워드 사용이 3건에 불과한 점은, 거대 파일치고 부작용(side effect)을 의식적으로 통제했다는 신호다.

**3. 성능 최적화가 "문서상 주장"이 아니라 실제 코드로 존재한다.**
parquet 디스크 캐시 15곳, TTL/타임스탬프 기반 캐시 161곳, `_RESULT` RAM 캐시·`_disk_read/write` 14곳, `asyncio.gather` 5곳, NumPy 벡터화 88곳, `sliding_window_view`(프랙탈 매칭) 2곳, `go.Surface`/`Scatter3d` WebGL 5곳, DuckDB 9곳이 모두 실재한다. CLAUDE.md가 광고하는 M4 활용 포인트가 빈말이 아니다.

**4. 폴백(fallback) 설계가 실전적이다.**
DuckDB 미설치 시 `_screener_rows_pandas`(pyarrow)로 자동 강등(7390줄), KIS→네이버, Yahoo crumb 401 시 무인증 search API로 강등 등 "데이터가 없으면 화면이 비지 않게" 하는 방어가 일관적으로 깔려 있다. 개인용 앱의 가용성을 실제로 끌어올리는 부분이다.

**5. API 키 입력 경로가 단일하고 깔끔하다.**
키는 전부 `os.environ`(23회)을 통해서만 읽고, `load_dotenv`로 루트 `.env` + `api_documents/API.env`(override=False)를 로드한다. **메인 백엔드 코드에 하드코딩된 키는 0건.** 키 관리의 "입력" 측면은 모범적이다(노출 측면은 문제점 5 참조).

## 문제점 (심각도 순)

### 🔴 1. 자동화 테스트가 사실상 0 — 핵심 비즈니스 로직 검증 불가
- **근거**: 프로젝트 전체에서 비즈니스 로직 테스트 파일은 `test_ov_api_keys.py`(키 점검), `application_build/test_squircle.py`(아이콘), `_sheet_test.py` 셋뿐이다. 검색된 다른 모든 `test_*.py`는 벤더링된 KIS 라이브러리(`API_documents/KIS_github/.../​.venv`) 내부 파일이다. 퀀트 계산(Sharpe/Sortino/VaR/몬테카를로/CAPM 회귀), 캐시 일관성, API 파싱(`tomv` 스케일·EUC-KR 디코딩·KIS iscd 필터)에 대한 단위 테스트 커버리지는 **0%**.
- **실질적 위험**: changes 로그가 자인하듯(trap #11, #27) 데이터 스케일 버그("NVDA 511,248,232.0조")가 실제로 출시됐다가 사후 발견됐다. 테스트가 없으면 이런 회귀를 매번 사람이 눈으로 잡아야 하고, AI 에이전트가 "고쳤다"고 단언해도 객관적 게이트가 없다. 금융 수치는 한 자리 오차도 의사결정을 왜곡한다.
- **개선 방향**: 순수 함수부터 시작. `fmtMcap`, Sharpe/VaR/CAPM, `_kis_stock_news` iscd 필터, 스케일 변환(×10, 억/조) 같이 **네트워크 없이 입력→출력이 결정적인 함수**에 `pytest` 20~30개만 붙여도 회귀의 80%를 잡는다. 난이도 보통.

### 🔴 2. "py_compile OK = verified" 및 미검증 기능 누적
- **근거**: `_STATUS.md`에 `❓ 미검증` 25개, "visual pending" 19회, `⚠` 6개가 누적. changes 로그 status 분포는 verified 38 / partial 15 / unverified 5 / broken 1. CLAUDE.md §5가 "`py_compile` 단독은 항상 unverified"라고 못 박았는데도, 실제로는 컴파일/`test_client` 통과만으로 ✅ 처리되는 경향이 로그에 보인다.
- **실질적 위험**: Correction log가 증명한다 — changes_2는 "AI·스크리너 고침"이라 기록했으나 둘 다 HTTP 500/모듈 누락으로 깨져 있었고, changes_13은 "런치 플리커 고침"이 거짓, changes_28은 추론모델 처리가 "정반대"였다. **검증 없는 "fixed" 기록은 다음 세션 에이전트를 잘못된 전제 위에서 작업하게 만든다** — 이 프로젝트에서 가장 비싼 실패 모드라고 문서 스스로 명시(§2 Anti-"claimed-fixed").
- **개선 방향**: GUI 시각 검증이 필요한 항목은 솔직히 `unverified`로 두는 현재 원칙은 옳다. 다만 ❓ 25개가 줄지 않고 쌓이는 게 문제이므로, **"미검증 부채 목록"을 별도 섹션으로 빼서 세션마다 1~2개씩 시각 확인 후 소거**하는 루틴을 만들 것. 난이도 쉬움(문서 운영).

### 🟡 4. 문자열 앵커 기반 HTML 주입의 취약성
- **근거**: `_inject_*` 함수 7종이 `html.replace("</nav>", ...)`, `html.replace("<footer", ...)`, `html.count('class="tab-btn')` 같은 **문자열·정규식 앵커**로 빌드된 HTML을 사후 수술한다(2162줄 `_inject_m4_tab` 등). 전체 `.replace()` 86회, HTML 대상 `re.sub/search` 46회. 탭 인덱스는 `class="tab-btn"` 문자열 개수를 세어 맞춘다.
- **실질적 위험**: 원본 빌더(`dashboard.py`, `company_report_ver2.py`)가 `</nav>`나 클래스명을 조금만 바꿔도 주입이 **조용히 실패**(아무 에러 없이 탭이 안 붙음)한다. 이건 컴파일·`test_client`로는 안 잡히는 종류라 "verified"를 통과해버린다. 설계 원칙(원본 비파괴)은 합리적이나 결합도가 숨어 있다.
- **개선 방향**: 앵커가 사라졌을 때 **명시적으로 실패하게** 만들 것 — `_inject_m4_tab`처럼 `if "</nav>" not in html: return html`로 조용히 무시하지 말고, 최소한 로그 경고를 남기거나(개발 모드) 주입 성공 카운트를 검증하는 가벼운 assert를 추가. 근본 해결은 원본 빌더에 `<!--KMKT_M4_SLOT-->` 같은 **전용 주석 앵커**를 심는 것. 난이도 쉬움.

### 🟡 6. XSS 방어가 escape 없이 "신뢰 가정"에 의존
- **근거**: 메인 백엔드에 `markupsafe`/`html.escape`/`Markup()` 사용 0건. 외부 데이터를 HTML로 짜는 f-string이 24곳 — 예: `f'<a href="{home}" ...'`(5442줄, DART 기업 홈페이지 URL을 그대로 삽입), 종목명·뉴스 제목 등을 escape 없이 보간.
- **실질적 위험**: 데이터 소스(네이버·DART·KIS)가 악의적이지 않다는 전제라 즉각적 위협은 낮지만, **외부 문자열을 `href`/innerHTML에 무검증 삽입**하는 패턴은, 만약 외부 응답에 `"><script>`나 `javascript:` URL이 섞이면 로컬 WebView 안에서 스크립트가 실행될 수 있다(로컬 앱이라 파일 접근 등 영향 범위가 더 넓을 수 있음). LLM이 생성한 마크다운을 innerHTML로 렌더하는 경로(`kmkt-md`)도 같은 계열 위험.
- **개선 방향**: 사용자/외부 텍스트를 HTML에 넣는 지점에 `html.escape()` 적용, `href`는 `http(s)://`로 시작하는지 화이트리스트 검증. 개인용 단일 사용자라 🟡로 분류하나 escape는 비용이 거의 0이라 적용 권장. 난이도 쉬움.

*이유*: 테스트를 못 만들어도, ❓ 25개를 세션당 1~2개씩 시각 확인해 ✅/❌로 확정하면 "선언≠실제" 간극이 줄어든다. AI 에이전트에게 "이번 세션은 ❓ 목록에서 N개를 직접 확인하라"고 시키는 것만으로 운영 가능. (문제점 2)

**3. 핵심 순수 함수에 pytest 20~30개 (난이도: 보통)**
*이유*: 금융 수치 회귀(스케일·계산 오류)는 이미 실제로 터졌다(`tomv` 버그). 네트워크 없는 결정적 함수(`fmtMcap`, VaR/Sharpe, iscd 필터)만 테스트해도 가장 비싼 버그 종류를 자동으로 막는다. AI 에이전트는 테스트 코드 생성을 잘한다 — "이 함수의 pytest를 써줘"가 적합한 작업. (문제점 1)

**4. HTML 템플릿 21개를 별도 파일로 추출 (난이도: 보통)**
*이유*: 로직을 1줄도 안 바꾸고 메인 파일을 ~8,000줄 줄여, 이후 모든 AI 작업의 정확도·속도를 끌어올린다. 모듈 분리(Blueprint)의 안전한 첫 단계. (문제점 3)

**5. `_inject_*` 주입 실패를 "조용히" 넘기지 말 것 (난이도: 쉬움)**
*이유*: 앵커가 깨지면 테스트가 통과해도 기능이 사라진다. 주입 성공 여부를 검증/로깅하게 바꾸면 #2의 "미검증" 리스크와 직접 연결된 침묵 실패를 표면화할 수 있다. (문제점 4)

**6. changes_history 두 디렉터리 통합 (난이도: 쉬움, 후순위)**
*이유*: 위험은 낮으나 에이전트 혼선의 잠재 원인. 키·테스트·검증이 정리된 뒤 정돈 차원에서. (문제점 7)

**7. 외부 텍스트 HTML escape (난이도: 쉬움, 상시)**
*이유*: 단일 사용자라 급하진 않지만, 새 화면을 만들 때마다 `html.escape()`를 습관화하면 누적 위험이 0에 수렴한다. (문제점 6)

> 의도적으로 **뒤로 미룬 것**: 13,000줄 단일 파일의 전면 재설계, 모듈 완전 분리, 코드 서명/공증. 모두 "있으면 좋지만" 1인 개발자에게 비용 대비 효용이 낮고, 위 1~5가 끝나기 전엔 오히려 리스크다.
